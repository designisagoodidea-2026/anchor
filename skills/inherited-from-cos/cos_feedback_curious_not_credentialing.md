**Rule:** When drafting Jason's voice for anything evaluator-facing — questions for interviewers, openers, bridge sentences, cover letters, recruiter outreach, networking-message specifics — lead with curiosity, not assertion. Don't credential. Don't position above onboarding. Don't ask trap-shaped questions.

**Why:** Established 2026-05-29 after Jason rewrote question #1 in the Collin VP prep workspace. The original draft sounded like a candidate grading the interviewer; his rewrite sounds like an experienced operator who is interested and wants to learn. Both can carry substance — only the second invites a real answer.

Jason's exact words: *"Here's a lightweight re-write on the tone / voice for the Questions for Collin, but applicable to all writing in my voice."*

## The patterns (diff-by-diff)

### Pattern 1 — Drop credentialing words

| Don't write | Do write |
|---|---|
| *"the design org's actual role"* (when "actual" is doing credentialing work) | *"the design org's role"* — or keep "actual" only when it's substantively earning its place |
| *"I've already done the substantive homework"* | *"I've already done my homework"* |
| *"the substantive observation that stuck"* | *"the observation that stuck"* |
| *"the real work"* / *"the real problem"* (when "real" is implicit credentialing) | drop the qualifier; the work is the work |

**Why:** these qualifiers signal that Jason is performing seriousness rather than carrying it. The reader notices the qualifier, not the substance.

### Pattern 2 — Drop "I'd already be operating / not onboarding" positioning

| Don't write | Do write |
|---|---|
| *"the platform-vs-product trade-off is where I'd be operating, not somewhere I'd need to be onboarded into"* | *"the platform-vs-product trade-off is interesting operational territory. I'd be interested in understanding the levers (e.g. cost/benefit, risk/reward, authority, jobs to be done...) at Superhuman today."* |
| *"the work I'd be carrying"* | *"the work that's interesting to me"* |
| *"I wouldn't need to be onboarded into"* | drop entirely — implies the speaker is testing the role, not curious about it |

**Why:** the first pattern says "I'm above this." The second says "I'm interested in this." Both can be true; only the second is the right thing to say in a question.

### Pattern 3 — Drop reductive trap-shaped question structures

| Don't write | Do write |
|---|---|
| *"Where does design have a vote, where does it have a say, and where is it being asked after?"* (triplet trap — the third position is the wrong answer; question reads as a test) | *"Where does design influence the decision?"* — open, invites a real answer including "it depends" or "less than I'd like" |
| Any three-position question where one position is the obvious "wrong" answer | one open question that lets the interviewer choose their own framing |

**Why:** triplet questions where one position is clearly negative read as gotchas. The interviewer hears the trap. Open questions invite honest answers, including the negative case if it's true.

### Pattern 4 — Warmer connectors

| Don't write | Do write |
|---|---|
| *"Kevin volunteered X on the call"* (implies it wasn't asked-for, slightly clinical) | *"Kevin mentioned X on the call"* (just describes) |
| *"From your seat"* (operator jargon, hierarchical) | *"From your perspective"* (warmer) |
| *"I've been turning that frame over since"* (performative, dense) | *"I've been thinking more about that model and I'm curious"* (plainer, warmer) |
| *"I've been sitting with X"* | *"I've been thinking about X"* |

**Why:** the first column in each row reads as performance language. The second column does the same work without the performance.

### Pattern 5 — Ground curiosity with concrete examples

| Don't write | Do write |
|---|---|
| *"I'd want to understand the operating context."* | *"I'd be interested in understanding the levers — cost/benefit, risk/reward, authority, jobs to be done."* |
| *"Tell me more about how the team works."* | *"What do the planning rhythms look like — quarterly review, weekly priority calls, how decisions get reopened or not?"* |
| Abstract "I'd want to learn" framings | A specific list (3–5 concrete handholds) the interviewer can pick from |

**Why:** abstract curiosity is performative. Concrete curiosity gives the interviewer something to grab. The list also signals what Jason already considers material — without crediting him for it.

### Pattern 6 — Replace "vote / say / asked-after" with plainer language

The vote/say/asked-after frame got burned in this round; don't repeat it. Plainer substitutes:

- *"Where does design have influence over the decision?"*
- *"Where is design at the table when this gets decided?"*
- *"Who weighs in, and when?"*

## The unifying principle

**Lead with what's interesting to learn, not with what's been figured out.**

Jason's read is sharper than most candidates' — that's already in the room from the prep, the through-line, the case stories. The questions and openers don't need to demonstrate it; they need to *invite the conversation that uses it*. Curiosity is the carrier; assertion is the noise.

Test before sending:
- Does any phrase in this draft sound like "I'm testing whether you measure up"? → rewrite.
- Does any phrase position Jason as "already there" or "wouldn't need onboarding"? → rewrite.
- Is any question shaped so one of the available answers is clearly wrong? → reshape.
- Could a warmer connector do the same work? → swap.
- Is there a place where a concrete list (3–5 items) would replace an abstract claim? → add the list.

## Applies to

- **All writing in Jason's voice** — Jason explicitly named the universality.
- **Questions for interviewers** (the immediate trigger).
- **Openers** ("tell me about yourself" / through-line answers).
- **Bridge sentences** in story banks and mock Q&A.
- **Cover letter close** especially — the close is where credentialing language tends to creep in.
- **Recruiter outreach** opening lines.
- **Networking-message specifics** when describing past work.
- **Thank-you notes** post-interview.

## Does NOT apply to

- **Internal memory dossiers** — descriptive language, not Jason's voice in performance mode. Case dossiers and intel files can use the trap-shaped framings descriptively when they're the actual analytic content (e.g., "Molly's criteria are X / Y / Z").
- **Pressure-testing or critique output to Jason** — that's a different mode where directness over warmth is the right setting.

## Examples worth preserving

**Question #1 — before (the burn):**
> *"Kevin volunteered 'Compound Startup, embedded not pooled' on our HM call. I've been turning that frame over since. From your seat, what's the design org's actual role in the platform-vs-product trade-off? Where does design have a vote, where does it have a say, and where is it being asked after?"*

**Question #1 — after (Jason's rewrite):**
> *"Kevin mentioned that Superhuman operates as a 'Compound Startup' where resources are 'embedded not pooled'. I've been thinking more about that model and I'm curious — from your perspective what's the design org's role in the platform-vs-product trade-off? Where does design have influence over the decision?"*

Same substance. Different posture. The second invites a real answer; the first audits one.
